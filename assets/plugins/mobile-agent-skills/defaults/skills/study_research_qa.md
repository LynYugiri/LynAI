# 开放问题检索综述

当用户问的问题需要联网检索、多源验证、汇总成综述（带引用源）时使用。本 skill 是"澄清问题→多源检索→综合→沉淀笔记"的流程。

## 适用场景

- 事实型问题（"X 是什么时候发生的"）。
- 比较型问题（"A 和 B 哪个更适合 ..."）。
- 调研型问题（"用什么方案能解决 ..."）。
- 需要给引用源的场景。

## 原则

- 不杜撰引用，所有引用必须有可访问的 URL 或明确来源标注。
- 多源验证：核心事实至少 2 个独立来源；单源信息明确标 `单一来源`。
- 综述里按置信度排序：高置信先、低置信标注保留意见。
- 复杂问题先用 `update_plan` 列检索子任务，避免漏关键点。
- 关键发现、来源、置信度写入 `update_agent_memory`。
- 沉淀综述用 `save_note`，存为带引用的 Markdown。

## 分流原则

- 用户问的事实可由主模型知识直接答：先答、标"截至训练数据日期"，用户要更新再检索。
- 用户问到训练截止后的事：直接走 `web_fetch` 检索最新数据。
- 用户要"带引用的"或"调研报告"：必走多源+综述+笔记沉淀。
- 用户只要一句话答案：跳过综述，给答案+1 个引用即可。

## 调用流程

```
1. 澄清问题边界
   └── 主模型归纳"要回答什么/不要回答什么"
2. 列检索子任务
   └── update_plan 多条
3. 检索
   ├── 主模型直接答（已知事实）
   └── web_fetch 抓外部页/搜索结果
4. 验证
   └── 核心事实至少 2 源；不足标"单一来源"
5. 综述
   └── 按置信度排序，每条带 `source`/`confidence`
6. 沉淀（用户明确要求）
   └── save_note 综述+引用
```

## 工具调用示例

`web_fetch`（抓取外部页）：

```json
{
  "name": "web_fetch",
  "arguments": {
    "url": "https://example.com/article",
    "maxChars": 12000
  }
}
```

`update_plan`（检索子任务）：

```json
{
  "name": "update_plan",
  "arguments": {
    "items": [
      { "id": "research_1", "status": "in_progress", "summary": "查 X 的官方定义" },
      { "id": "research_2", "status": "pending", "summary": "查 X 的对比评测" },
      { "id": "research_3", "status": "pending", "summary": "查 X 的常见踩坑" }
    ]
  }
}
```

`save_note`（综述沉淀）：

```json
{
  "name": "save_note",
  "arguments": {
    "title": "调研-X 方案对比-2025-07-04",
    "content": "## 问题\n...\n## 主要发现\n1. ...\n## 引用\n1. [来源1](https://...) 置信度 0.9\n2. [来源2](https://...) 置信度 0.6\n## 结论\n...",
    "folderId": "调研回顾"
  }
}
```

## 返回约定

- 综述用清单：`发现 → 引用源 → 置信度`。
- 不可访问/解析失败的引用标"未验证"，不计入综述证据。
- 单一来源信息明确标 `单一来源`，避免误导。

错误码：

| 码 | 说明 |
|---|---|
| `ok` | 成功 |
| `fetch_failed` | URL 抓取失败 |
| `source_insufficient` | 来源不足 2 个 |
| `save_failed` | 笔记保存失败 |

## 失败处理

- `fetch_failed`：换备用 URL 或换关键词重新检索；持续失败标"未验证"。
- `source_insufficient`：显式告知用户"仅找到单一来源，结论可能不稳"。
